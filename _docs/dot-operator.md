The dot is Eliot's most characteristic operator. It turns nested calls inside-out into a left-to-right
chain that reads in run order — and it shapes how every function in the language is designed.
{: .docs-lead}

## The dot is a function too

Like every operator, `.` is an ordinary definition. Simplified, it is:

```eliot
infix left below apply
def .[A, B](a: A, f: Function[A, B]): B = f(a)
```

So `a.f` is just `f(a)`. It is **positional, not member-based** — there is no notion of "methods on an
object". `a.f` works for *any* function `f` that can take `a`. That one fact is why field accessors,
library functions, and your own functions all dot-chain uniformly.

## Reading a chain

Because `.` is left-associative and binds just below application, a chain unfolds left to right:

```eliot
subject.f(a).g(b)     //  ==  g(b, f(a, subject))
```

Compare the two spellings. The nested form `g(b, f(a, subject))` reads inside-out — you find
`subject` in the middle and work outward. The dot form reads in the order things happen: start with
`subject`, apply `f(a)`, then `g(b)`. Here is the real example, `DotOperator.els`:

```eliot
import eliot.effect.Console

data Box[A](content: A)

def filter[A](expr: String, box: Box[A]): Box[A] = box

def map[A, B](f: A => B, box: Box[A]): Box[B] = Box(f(box.content))

def flatMap[A, B](f: A => Box[B], box: Box[A]): Box[B] = f(box.content)

def as[A, B](b: B, box: Box[A]): Box[B] = box.map(_ -> b)

def logic: Box[String] = Box("Hello").filter("Expr").map(_ -> "Earth!").as("World!")

def main: IO[Unit] = printLine(logic.content)
```

Notice `box.content` inside the definitions — field access is the same operator; `content` is just the
one-argument accessor generated by `data Box`. And `logic.content` at the end reads the final result.

## Design consequence: subject last

For `subject.f(a)` to mean `f(a, subject)`, the subject must be `f`'s **last** parameter. Look back at
the signatures above:

```eliot
def map[A, B](f: A => B, box: Box[A]): Box[B]      // the box is LAST
def filter[A](expr: String, box: Box[A]): Box[A]   // the box is LAST
```

This is the **house style for every function that has a natural subject**, not just the ones you
expect to chain. It's the opposite of the "receiver comes first" habit from object-oriented
languages, so it's worth checking on every new signature you write: configuration and auxiliary
arguments go first, and the thing being transformed goes last.

Once functions are designed this way, partial application (from
[Values & functions]({{ '/docs/values-and-functions/' | relative_url }})) clicks into place: `map(f)`
is a `Box[A] => Box[B]`, ready to be the right-hand side of a dot.

## Long chains across lines

A chain that outgrows one line breaks with a **leading dot**, which merges with the line above
(that's the line-joining rule from [Blocks]({{ '/docs/blocks/' | relative_url }})):

```eliot
def logic: Box[String] = Box("Hello")
  .filter("Expr")
  .map(_ -> "Earth!")
  .as("World!")
```

This is one expression, formatted for reading — each step on its own line, in order.

## The juxtaposition gotcha

There is one trap, and it catches everyone once. Plain application by juxtaposition (`f x`) binds
**tighter than the dot** — tighter than every infix operator, in fact. So:

```eliot
printLine msg.content     // parses as  content(printLine(msg))  — a type error!
printLine(msg.content)    // what you meant: print the content of msg
```

When an argument carries its own dot chain (or any infix operator), wrap it in parentheses so it
isn't swallowed by juxtaposition. The rule of thumb: prefer the adjacent-paren call form `f(x)`, and
reserve bare juxtaposition for clean single-argument, operator-flavored calls like `forever { … }`.

## Direct style, the through-line

The dot operator is one half of Eliot's readability story; **direct style** is the other. You write
effectful steps as if they were ordinary values — `printLine(readLine)` — and let the compiler insert
the sequencing. You never hand-write `flatMap` towers in application code. Blocks, dot chains, and the
effect system are all designed so the code you write is the code you'd sketch on a whiteboard.
